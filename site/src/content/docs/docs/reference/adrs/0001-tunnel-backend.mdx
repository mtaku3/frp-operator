title: "ADR 0001: Tunnel backend selection"

**Status:** Accepted — 2026-04-23

## Context

The operator provisions cloud VPSs as tunnel exit nodes that expose in-cluster
services on public TCP ports (inlets-operator-style L4 port-forward).
Requirements:

1. Dynamic add/remove of tunnels on a live server without restart, driven by
   the K8s reconcile loop.
2. 1:N tunnels per VPS (multi-tenant).
3. No custom kernel dependency on the VPS.
4. Clients may also be set up manually outside the operator to expose
   non-Kubernetes services.

## Decision

Use **FRP** (`fatedier/frp`) as the tunnel backend.

## Alternatives considered

| Backend | Rejected because |
|---|---|
| WireGuard | L3 VPN, not port-forward — mismatched semantics; requires kernel module on VPS; would need a DNAT layer to expose per-tenant ports. |
| Chisel | No server-side admin API; tunnels are client-initiated only, so the operator cannot mutate state on a running server. |
| rathole | Config-file driven; adding a tunnel requires rewriting server TOML + reload. Best raw throughput but loses the no-restart property. |
| bore | Fixed 15000–19999 port range, no auth model beyond a shared secret, ~12 s reconnect time. Too rigid for multi-tenant use. |

## Rationale

- FRP has a REST admin API (`/api/config`, reload) that maps onto a reconcile
  loop. Of the benchmarked backends, only FRP and WireGuard support true
  dynamic add, and WireGuard's L3 semantics don't fit the usecase.
- L4/L7 proxy semantics (`tcp`, `udp`, `http`, `https`, `stcp`) match the
  "expose service on port X" model.
- Per-client tokens support multi-tenant auth out of the box.
- Benchmarks: mid-pack throughput (2.7–3.2 Gbps, within ~15% of the leader),
  best HTTP-large-body rps among tunnel backends, fastest reconnect (0.4 s).
- Weak spots are acceptable: highest per-peer memory (~122 KiB) and highest
  add-disruption dip (16%), but both stay within operator-scale budgets.
- Standalone `frpc` is trivial to run outside Kubernetes, preserving the
  manual-client use case.

## Consequences

### Port reservation on each VPS

- **`bindPort`** (default 7000, TCP) — control channel; every `frpc` connects
  here.
- **`webServer.port`** (default 7500, TCP) — admin REST API. Operator-only;
  bind to a private interface and firewall off from the public.
- **`allowPorts`** range (operator-configured, e.g. 20000–29999, TCP) — the
  tenant pool. The operator allocates a free port from this range per tunnel
  and writes it into the client's proxy config.
- **`vhostHTTPPort` / `vhostHTTPSPort`** (default 80/443, TCP) — optional, only
  if name-based HTTP multiplexing is enabled.
- **`kcpBindPort` / `quicBindPort`** (UDP) — optional, only if those
  transports are enabled.

### Memory footprint

Baseline `frps` idle is ~4.5 MB RSS, plus ~122 KiB per idle tunnel. Active
traffic adds a few tens of KB of buffer per live TCP connection (tunable via
`transport.poolCount`). Projected idle RSS:

| Tunnels | Idle RSS (est.) |
|---|--:|
| 10 | ~6 MB |
| 100 | ~17 MB |
| 1,000 | ~127 MB |
| 5,000 | ~620 MB |

The practical per-VPS density limit is port-range exhaustion or CPU at
line-rate traffic, not `frps` RAM.

### Operational

- The operator owns `frps` lifecycle on each VPS and mutates its config via
  the admin API.
- Coupled to FRP's protocol evolution; upstream version pins are managed in
  the operator.
- **Version pinning:** Currently pinned to v0.68.1 (latest stable as of 2026-04).
  Bumped from v0.65.0 during Phase 2 implementation for performance and
  stability improvements.
